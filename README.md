# Doc Transformer

Converts a Markdown document into a styled Word `.docx` file by applying the styles, page layout, header/footer, and images from an existing Word template.

Architecture mirrors AWS:
- **React** frontend (Vite) — replaces a static S3 website
- **Flask** server — replaces API Gateway
- **`lambda_handler.py`** — the actual Lambda function, called inline
- **`db.py`** (JSON file) — replaces DynamoDB

---

## Quick Start

### 1. Backend

```bash
cd backend
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

Generate the sample Word template (requires Pillow for the logo image):

```bash
python create_template.py
# → templates/corporate_template.docx
```

Start the API server:

```bash
python server.py
# → http://localhost:5000
```

### 2. Frontend

```bash
cd frontend
npm install
npm run dev
# → http://localhost:3000
```

Open `http://localhost:3000`, upload `templates/corporate_template.docx`, click **Load sample** to fill the editor, then click **Generate Document**.

## How It Works

### Template extraction

The processor opens the `.docx` template with `python-docx`, which loads:

| What | How preserved |
|------|---------------|
| Page size & margins | `w:sectPr` element kept intact |
| All paragraph/character styles | Styles XML part is untouched |
| Header (logo, company name) | Header relationships kept |
| Footer (page numbers, text) | Footer relationships kept |
| Images in header/footer | Part references kept |
| Theme colours | `theme/theme1.xml` kept |

The body paragraphs are cleared; headers, footers, styles, and section properties are left in place.

### Markdown → Word mapping

| Markdown | Word style |
|----------|-----------|
| `# H1` | `Heading 1` |
| `## H2` | `Heading 2` |
| `### H3` | `Heading 3` |
| `#### H4` | `Heading 4` |
| Regular paragraph | `Body Text` → `Normal` |
| `> blockquote` | `Intense Quote` → `Quote` → `Normal` |
| ` ``` code ``` ` | `Code` → monospace fallback |
| `- item` | `List Bullet` |
| `1. item` | `List Number` |
| `\|table\|` | `Table Grid` |
| `---` | Paragraph bottom border |
| `**bold**` | bold run |
| `*italic*` | italic run |
| `~~strike~~` | strike run |
| `` `code` `` | Courier New run |
| `[text](url)` | hyperlink |
| `![alt](src)` | inline picture (data URI or HTTPS) |

Style names fall back gracefully: if `Heading 2` does not exist in the template, `Normal` is used.

### API

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/api/process` | Upload template + markdown → generate `.docx` |
| `GET` | `/api/documents` | List generated documents |
| `GET` | `/api/documents/:id` | Get document metadata |
| `GET` | `/api/documents/:id/download` | Download the `.docx` |
| `DELETE` | `/api/documents/:id` | Delete record + file |
| `GET` | `/api/health` | Health check |

`POST /api/process` accepts `multipart/form-data`:
- `template` — `.docx` file
- `markdown` — plain text

## Deploying to Real AWS

To deploy to actual AWS, swap out:

| Emulated component | Real AWS service |
|--------------------|-----------------|
| `server.py` (Flask) | API Gateway + Lambda |
| `db.py` (JSON file) | DynamoDB table |
| `uploads/outputs/` | S3 bucket |
| `frontend/` (Vite dev) | S3 static website + CloudFront |

The `lambda_handler.py` file is already structured for Lambda deployment — its `event` and `context` parameters follow the API Gateway proxy integration format exactly. Package it with `python-docx`, `markdown`, and `beautifulsoup4` in a Lambda layer or container image.
